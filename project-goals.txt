**Project Overview & Design Document**

## 1. Introduction
### 1.1 Purpose
The purpose of this project is to develop a flexible, multi-tenant estimating software designed for professionals who estimate jobs without detailed prints. Unlike traditional "take-off" software, this tool is intended for small contractors and individuals needing a dynamic, customizable estimation system.

### 1.2 Target Audience
- Small contractors and individual estimators.
- Businesses needing a lightweight estimating tool.
- Users who prefer an open-source, self-hosted solution.

### 1.3 Key Problems Solved
- **Dynamic Estimation:** Instead of requiring predefined components for every variation (e.g., different wire lengths), this system allows unit-based cost and labor calculations.
- **Editable Packages:** Users can modify packages at the estimate level without affecting base components.
- **Open-Source:** Enables community contributions and broad accessibility.

## 2. Core Features
### 2.1 Customization
- Users can define their own labor and material costs.
- Packages and components can be modified per estimate.
- Labor rates (both cost and price) are set per hour through the settings interface.

### 2.2 User Roles
- **Admin:** Full system access.
- **Estimator:** Can create and modify estimates.
- **Manager:** Oversees estimations but with limited admin access.

### 2.3 Multi-Tenancy
- Self-hosted instances support multiple departments or businesses without cross-over.
- Complete tenant isolation with middleware and query scoping.

### 2.3.1 Multi-Tenancy Implementation
- Row-level multi-tenancy with tenant_id on all relevant tables.
- Middleware enforces tenant isolation throughout the application.
- Query scoping prevents data leakage between tenants.

### 2.4 Data Structure
- **Items:** To be implemented in all three forms:
  1. Direct materials (attached directly to estimates)
  2. As part of assemblies
  3. As base templates for future use
  
  Each item includes:
  - Labor:
    - Time in minutes (stored in database)
    - Time displayed in hours (in views/exports)
    - Labor cost (company's cost per hour in database)
    - Labor price (customer billing rate per hour in database)
  - Materials:
    - Material cost (company's purchase cost)
    - Material price (customer billing price)
    - Quantity
    - Unit of measure

- **Assemblies:** Groups of items with predefined quantities
- **Packages:** Collections of assemblies that form a complete estimate
  - Dynamic addition of assemblies to packages
  - Proper cost and charge calculations
- **Labor Rates:** Customizable labor costs per unit
- **Status Tracking:** Estimates can move through statuses (Draft, Submitted, Approved, etc.)
- **Version Control:** Planned for future implementation

### 2.4.1 Package and Assembly Structure
- Complete copying of packages and assemblies when added to estimates
- Independent modifications of items within estimates
- Base templates remain unaffected by estimate-level changes
- Dynamic assembly addition to existing packages

### 2.4.2 Calculations and Data Handling
- **Time Conversions:**
  - Database stores labor time in minutes for precision
  - UI displays time in hours (minutes/60) with configurable decimal places
  - Calculations use minutes internally for accuracy
  
- **Cost and Price Calculations:**
  - Labor costs calculated as: (minutes/60) * labor_cost_rate
  - Labor price calculated as: (minutes/60) * labor_price_rate
  - Material costs calculated as: quantity * unit_material_cost
  - Material price calculated as: quantity * unit_material_price
  - Assembly totals sum all contained item costs and prices
  - Package totals sum all assembly totals plus direct items

- **Rounding and Precision:**
  - Money values stored with 4 decimal places for accurate calculations
  - Display formatting configurable per tenant:
    - Decimal places for hours display
    - Decimal places for money display
    - Currency symbol and position
    - Thousands separator preference

### 2.5 Workflow
1. Create new estimate
2. Add direct materials as needed
3. Select and add predefined packages
4. Add additional assemblies to packages or estimates
5. Modify items within the estimate
6. Generate exportable estimates

### 2.6 Export & Collaboration
- **Exports:** Functionality for exporting estimates
- **Collaboration:** Planned for future versions

### 2.6.1 Export Formats
- PDF and CSV exports
- Customizable export templates

### 2.7 User Onboarding
- Streamlined registration and setup process
- Company/tenant setup workflow
- Basic system navigation and usage guidance

## 4. Tech Stack
- **Backend:** Laravel 12 (PHP Framework)
- **Frontend:** 
  - Livewire 3 with Tailwind CSS
  - Flux UI (free version) for core components
  - Chart.js for data visualization
  - SimpleMDE for Markdown editing
- **Database:** MySQL
- **Version Control:** Git with GitHub repository
- **Development Environment:** Laravel Sail (Docker-based)
- **Hosting:** Self-hosted with Docker support
- **Cache & Queue:** Redis

### 4.1 Development Environment
The project uses Laravel Sail for development, which provides:
- Containerized development environment using Docker
- Pre-configured services (PHP, MySQL, Redis, etc.)
- Consistent development experience across different machines
- Easy setup and configuration

### 4.2 Recommended Packages
The following open-source packages will be utilized to implement core functionality:

#### Authentication & Multi-Tenancy
- **Laravel Jetstream with Flux UI (Free Version)**: Provides authentication, team management, and profile management
  - Team functionality will serve as the foundation for tenant organizations
  - Built-in two-factor authentication for security
  - Profile photo management and account settings
  - Modern Tailwind CSS UI components using Flux's free component library:
    - Buttons, dropdowns, and modals
    - Form inputs and select components
    - Checkbox and radio groups
    - Switch toggles
    - Basic layouts and navigation
  - Alternative solutions for advanced features:
    - SimpleMDE for rich text editing (Markdown-based)
    - Chart.js for data visualization
    - Native HTML5 date inputs
    - Traditional navigation patterns
  - Livewire-powered dynamic interfaces

- **Spatie Laravel-Permission**: Role and permission management
  - Define roles (Admin, Estimator, Manager)
  - Granular permissions for system actions
  - Integration with multi-tenant architecture

- **Spatie Laravel-Multitenancy**: Database isolation for multi-tenant architecture
  - Row-level tenant isolation
  - Automatic tenant identification
  - Query scoping to prevent data leakage

#### Data Management & Export
- **Barryvdh/laravel-dompdf**: PDF generation for estimate exports
  - Convert HTML templates to professional PDFs
  - Customizable templates for different estimate formats
  - Header/footer support for branding

- **Laravel Excel**: Spreadsheet imports and exports
  - Export estimates to Excel/CSV formats
  - Import material costs or labor rates
  - Batch processing capabilities

#### Versioning & Activity Tracking
- **Spatie Laravel-Activitylog**: Change tracking and audit trails
  - Log activities on estimates and other models
  - Track who made changes and when
  - Customizable logging for specific actions

#### Development & Testing
- **Laravel Debugbar**: Development debugging toolbar
  - Query monitoring
  - Request/response inspection
  - Performance profiling

- **Pest PHP**: Modern testing framework
  - Expressive test syntax
  - Better developer experience
  - Laravel-specific assertions

#### Implementation Strategy
These packages will be integrated following the phased approach:

**Phase 1:**
- Implement Jetstream for authentication and team management
- Set up Spatie Permission and Multitenancy for core security
- Build core UI components using Jetstream and Livewire
- Implement basic PDF export with laravel-dompdf

**Phase 2:**
- Add Laravel Excel for enhanced exports
- Implement Spatie Activitylog for tracking changes
- Expand Livewire components for more complex interfaces

**Phase 3:**
- Add API capabilities using Jetstream/Sanctum
- Implement more advanced features using specialized packages
- Optimize and refine the integration between packages

### 4.3 Testing Strategy
Testing will be a core component of the development process, with comprehensive test coverage required before any feature is considered complete. The following testing approach will be implemented:

#### Testing Tools & Frameworks
- **PHPUnit**: Core testing framework for unit and feature tests
- **Laravel Dusk**: Browser testing for critical user workflows
- **Pest PHP**: Modern testing framework for more expressive tests
- **Laravel Pint**: Code style enforcement
- **PHPStan**: Static analysis for early error detection
- **GitHub Actions**: CI/CD pipeline for automated testing

#### Types of Tests
1. **Unit Tests**
   - Test individual components in isolation
   - Focus on business logic and calculations
   - Ensure estimate calculations are accurate
   - Test tenant isolation mechanisms

2. **Feature Tests**
   - Test complete features and endpoints
   - Ensure proper authentication and authorization
   - Verify multi-tenant data isolation
   - Test estimate creation and modification workflows

3. **Browser Tests (with Laravel Dusk)**
   - End-to-end testing of critical user journeys
   - Test the complete estimation workflow
   - Verify UI components function correctly
   - Test responsive design and accessibility

4. **Static Analysis**
   - Enforce code quality standards
   - Identify potential bugs before runtime
   - Ensure consistent code style

#### Testing Requirements
- Minimum 80% code coverage for all new features
- All calculations must have 100% test coverage
- Regression tests for any bugs fixed
- Performance tests for database queries to ensure scalability
- Security tests for authentication and tenant isolation

#### Testing Approach by Phase

**Phase 1:**
- Establish testing infrastructure and CI/CD pipeline
- Create base test cases for multi-tenancy and permissions
- Implement unit tests for core estimation logic
- Develop feature tests for authentication and basic workflows

**Phase 2:**
- Add browser tests for complete user journeys
- Implement performance testing for database queries
- Expand test coverage for export functionality
- Add tests for activity logging

**Phase 3:**
- Implement API testing
- Add tests for version control and collaboration features
- Develop comprehensive regression test suite
- Optimize test performance for faster CI/CD pipeline

#### Test-Driven Development
Where appropriate, a test-driven development (TDD) approach will be used:
1. Write tests that define the expected behavior
2. Implement the minimum code required to pass the tests
3. Refactor while maintaining passing tests

This approach will be particularly important for the estimation calculation logic to ensure accuracy and reliability.
